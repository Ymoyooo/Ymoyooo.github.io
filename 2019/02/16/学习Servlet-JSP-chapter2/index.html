模型-视图-控制器(MVC)设计模式


MVC全名是Model View Controller，是模型(model)－视图(view)－控制器(controller)的缩写，一种软件设计典范，用一种业务逻辑、数据、界面显示分离的方法组织代码，将业务逻辑聚集到一个部件里面，在改进和个性化定制界面及用户交互的同时，不需要重新编写业务逻辑。MVC被独特的发展起来用于映射传统的输入、处理和输出功能在一个逻辑的图形化用户界面的结构中。

简单的来说，MVC是一种做事习惯，一种行事准则。

MVC展开就是:M(model)-V(view)-C(control)

如何理解?

这么讲一个故事吧，一个老板，一个杀手，一个律师。某人以某种行为激怒了老板，老板找到了杀手把这个人干掉了，然后律师根据杀手作案的信息来找漏洞为老板辩护写出辩护书。

老板就是Control，外界输入了信息给他，他去寻找一个模型解决问题。

杀手就是Model，这个杀手代表着一个逻辑，他只负责处理老板指示的那个人，处理完就走，不用负责打官司(生成HTML)

律师就是View，他负责处理杀手留下来的信息，即经模型处理后的信息的表达。

这里给我们的思想就是各司其职，老板不用苦练如何杀人和找寻法律漏洞，他只要会调用就行。模型(杀手)只需要关注好自己的逻辑就行，视图(view)只要会利用好杀手留下的信息来生成页面即可。

控制器:

从请求获得用户输入，并明确这些输入对模型有什么影响。告诉模型自行更新，并且让视图(JSP)能得到新的模型状态。

视图:

负责表示方面。它从控制器得到模型的状态(不过不是直接得到;控制器会把模型数据放在试图能找到的一个地方)。另外视图还要获得用户的输入，并交给控制器

模型:

包含具体的业务逻辑和状态。知道用什么规则来得到或更新状态。系统中唯一可以和数据库通信的部分。

MVC的优缺点一览：

优点：

  1. 耦合性低。 不至于牵一发而动全身。
  2. 重用性高。 模型可以被多个视图共享，别的应用也可以享受到这个服务。
  3. 部署快，生命周期成本低。编程人员各司其职，java的专注写java，界面程序员专注写界面。
  4. 可维护性高。 分离视图层和业务逻辑层也使得WEB应用更易于维护和修改。

缺点：

  1. 调试困难。 构件数量多，而且每个都要经过调试。
  2. 不适合小型、中等规模的应用程序。 MVC的优点在小型系统上体现不出来，反而使编写变得更加麻烦。
  3. 增加了系统结构和实现的复杂性。
  4. 视图与控制器间的过于紧密的连接并且降低了视图对模型数据的访问。
